2 — Root Causes: Slow Dashboard Loading
Diagnosed from code patterns and architecture — these are the performance killers
CRITICAL

No API Response Caching — Every Request Hits the DB

Market KPIs, inventory summaries, and trend data are re-queried from PostgreSQL on every single page load. With no Redis/in-memory cache layer, each dashboard tab triggers multiple full-table scans. At scale, this explodes load time to 3–8 seconds.

CRITICAL

LLM Runs Synchronously — Blocks the Entire API Thread

The Ollama Qwen2.5:7b call is made inline, synchronously in the FastAPI route handler. Since Qwen2.5:7b takes 5–30s to respond, the entire Python thread is blocked during inference. No streaming, no async, no background task queue — this directly makes the AI Advisor and any chat-related routes unusable under load.

CRITICAL

ML Models Loaded On-Demand (No Warm State)

The price_predictor, risk_scorer, demand_forecaster, and buyer_matcher pickle/joblib files are likely loaded on each request or on cold start. No singleton pattern = repeated deserialization on every prediction API call.

HIGH

No Database Connection Pooling

db.py likely creates a new SQLAlchemy engine/connection per request. Without connection pooling (min 5, max 20 connections) every API call pays a TCP handshake + auth cost to PostgreSQL. Under concurrent users this becomes a bottleneck immediately.

HIGH

Frontend Fetches All Dashboards on Mount

React frontend likely loads all 6 dashboard data sources on initial mount using parallel useEffect hooks. Even tabs that aren't visible trigger network requests. No lazy loading, no suspense boundaries, no route-based code splitting.

MED

No Database Indexes on Filter Columns

Inventory queries filter by make, model, risk_flag, year — none of these are likely indexed. Full sequential table scans on every filter operation. With 10k+ rows this becomes noticeable immediately.

MED

No ETL Materialized Views / Pre-Aggregation

Market trends and KPI aggregations are computed at query time. These should be materialized views refreshed on schedule (every 6–24 hrs), not recomputed on every API call.

✅ Fix 1: Redis Cache Decorator (API Layer)

# api/cache.py import redis.asyncio as redis import json, functools r = redis.from_url("redis://localhost:6379") def cache(ttl: int = 300): def decorator(func): @functools.wraps(func) async def wrapper(*args, **kwargs): key = f"cache:{func.__name__}:{kwargs}" cached = await r.get(key) if cached: return json.loads(cached) result = await func(*args, **kwargs) await r.setex(key, ttl, json.dumps(result)) return result return wrapper return decorator # Usage in routes: @router.get("/api/market/kpis") @cache(ttl=300) # 5 min cache async def get_market_kpis(): ...

✅ Fix 3: Async LLM Streaming (SSE)

# api/routes/chat.py from fastapi.responses import StreamingResponse import httpx, asyncio async def stream_ollama(prompt: str): async with httpx.AsyncClient(timeout=120) as client: async with client.stream("POST", "http://localhost:11434/api/generate", json={"model": "qwen2.5:7b", "prompt": prompt, "stream": True}) as resp: async for chunk in resp.aiter_lines(): yield f"data: {chunk}\n\n" @router.post("/api/chat") async def chat(req: ChatRequest): return StreamingResponse( stream_ollama(req.message), media_type="text/event-stream" )

⚠ LLM (Ollama / Qwen2.5:7b) — Special Consideration for Render

Render does not support GPU instances. Running Qwen2.5:7b on Render CPU-only will be very slow (60–120s inference). Two options:

  • Option A (Recommended for Render): Replace Ollama with Groq API (free, 500 req/day, Llama3 70B) — same capability, <2s response time. Or use OpenRouter for Qwen2.5 hosted.
  • Option B (Keep local feel): Run Ollama on a separate Railway or Modal.com GPU instance, expose as HTTP endpoint, point FastAPI at it.
